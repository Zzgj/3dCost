# 3D 打印成本控制平台 — 需求文档

> **项目**：拓竹 3D 打印机成本核算与报价平台
> **版本**：v1.0.0
> **更新**：2026-06-10
> **状态**：需求已完善，待实施

---

## 零、需求分析结论

当前需求方向清晰：平台不直接做“大而全 ERP”，而是围绕 3D 打印报价的核心链路，优先解决**真实到手价、自动核算、AI 调 API 录入、Web 手工兜底**这四件事。

本次完善重点：
- 明确 MVP 边界，避免一期实现被采购优化、小程序、自动抓取等探索项拖慢。
- 补齐成本计算规则，特别是快照、四舍五入、库存不足、完成锁定、子产品循环引用等边界。
- 补齐 API 统一约定，方便 AI 稳定调用，也方便前端复用同一套响应格式。
- 补齐数据一致性、备份、安全、部署和验收标准，使文档可以直接进入任务拆分和开发。
- 标记待确认问题，避免实现时靠临时判断造成返工。

### MVP 成功定义

一期成功不以页面数量为目标，而以主链路跑通为目标：

1. 建立耗材、零件、供应商、机器、成本参数等基础数据。
2. 录入采购批次后，系统自动维护库存与加权平均到手价。
3. AI 或 Web 创建打印件后，系统立即返回打印件成本。
4. AI 或 Web 创建产品 BOM 后，系统立即返回总成本、成本明细和建议报价。
5. 用户可手动触发库存扣减、生成报价快照，并在 Web 查看/修正/导出。

### 明确不做

以下内容不进入一期：
- 平台内置大模型、保存 AI 密钥、内置 MakerWorld 抓取器。
- 自动库存扣减、自动下单、采购最优解、满减/阶梯价模拟。
- 多用户权限体系、客户 CRM、订单管理、发票/收款管理。
- 云端部署、小程序上架、公开互联网访问。
- 批量打印费用自动分摊、复杂工艺排程、设备实时监控。

## 一、项目背景

购入拓竹 3D 打印机后，需要一个工具来精确核算每一件打印产品的成本，并据此给客户报价。

核心痛点：一件产品的成本不只是"耗材克重 × 单价"那么简单——还包括打印耗时电费、机器折旧、后处理（打磨/喷漆/组装），以及像"排风扇"这类需要外购电路板、电源等零件的复合产品。更关键的是：**零件成本要用真实到手价（含运费）**。手工算账既慢又容易漏项。

### 目标

做一个**单人自用**、**本地/局域网运行**、**前后端分离**的成本控制平台。它是一个**数据中心 + 计算引擎**，由 AI（Codex/Claude）作为主要操作员：

1. 在 MakerWorld 找到模型 → 发链接给 AI："帮我录入这个项目"
2. AI 抓取链接读出盘数/耗时/克重/喷嘴/多色用量 → **调平台 API 写入打印件**
3. 平台后台**自动触发成本计算**（结合库存里每种料的加权平均到手价）→ 返回给 AI
4. AI 告诉你："已录入，这个打印件成本 10.8 元"
5. 你补充其他零件/后处理 → AI 继续调 API 创建产品 → 平台算出总成本 → AI 告诉你"总成本 65.8 元，建议报价 105 元"
6. 你随时可打开 **Web 前端**：可视化查看所有产品/库存/报价历史/统计，也能手工录入/修正数据

架构上为后续上架**微信小程序**预留同一套后端 API。

> **核心设计**：AI 是主力操作员（调 API 录入/查询），Web 是管理界面（可视化 + 手工兜底）。平台提供完整 REST API + 后台自动成本计算，由一套轻量仓储（加权平均到手价）+ 供应商管理地基支撑。

## 二、关键决策

| 维度 | 决策 |
|------|------|
| 使用规模 | 单人自用（无需多账号/权限，保留单用户登录占位） |
| 部署 | 先本地/局域网；后续上云再加 HTTPS+备案（小程序前置条件） |
| 端 | Web 优先，前后端分离，后端 API 复用给未来小程序 |
| 后端 | Python + FastAPI |
| 前端 | Vue 3 + Vite + Element Plus |
| 数据库 | SQLite（本地零配置，后续可平滑迁 PostgreSQL） |
| **架构核心** | 平台 = REST API + 自动成本计算引擎；AI = 主力操作员；Web = 可视化 + 手工兜底 |
| **AI 交互** | AI 调平台 REST API 录入/查询数据；对话即可完成 90% 操作 |
| **成本计算** | 后台自动触发：录入打印件/产品时 API 返回即含成本明细 |
| **数据录入** | AI 调 API 为主（90%），Web 手工录入为辅（修正/补漏） |
| **克重/耗时/喷嘴** | AI 从 MakerWorld 链接抓取后调 API 写入 |
| **多色多材** | 按每种耗材分别录入用量、分别计成本（AMS） |
| 成本构成 | 打印基础成本 + 后处理 + 外购零件 + 废品损耗 |
| **零件取价** | 加权平均到手价：(历次采购含运费总额) / 总数量 |
| **仓储/供应商** | 一期核心地基。零件含到手价、库存、采购批次、供应商/采购链接；缺料一键跳链接补货 |
| **多机器** | 数据模型支持多台，一期 UI 只做单机器配置，预留扩展 |
| **耗材单位** | 采购按"卷"（含克/卷规格）→ 库存按"克" → 到手价按"克" |
| **零件单位** | 采购按"盒/瓶"（含换算比例）→ 库存按使用单位 → 到手价按使用单位 |
| 后处理人工 | 按工时计费（工时 × 人工单价） |
| 机器折旧+电费 | 录入购机价/寿命/功率/电价，系统自动折算每小时成本 |
| **复合产品** | 多层 BOM：多打印件 + 多零件 + 后处理工序，支持子产品嵌套（数据模型支持，一期 UI 不做） |
| **预估 vs 实际** | 两套并存：接单前「预估报价」+ 完工后「实际报价」，可对比 |
| **产品状态** | 极简：draft（草稿，可编辑）/ completed（已完成，锁定） |
| 客户报价 | 自定义加成比例（成本 × 倍率 或 成本 + 利润） |
| **报价输出** | 区分对内（全明细）/对外（仅最终价）；导出图片/PDF；历史与统计 |
| 库存扣减 | 手动触发（提供 API + 按钮），非自动 |
| **采购优化** | 本期不做（买几个最划算暂缓），列二期 |
| **费用分摊** | 本期不做（批量打印后处理分摊），列二期 |

## 三、业务概念与边界

### 核心对象

- **耗材 Material**：如 PLA/PETG/ABS，不同品牌、颜色、材质视为不同耗材 SKU。
- **零件 Part**：外购实物件，如电源、电路板、轴承、螺丝、胶水等，按使用单位计库存。
- **采购批次 Purchase**：一次采购记录，包含货款、运费、数量、供应商和链接，是计算真实到手价的唯一来源。
- **打印件 PrintItem**：一个可被产品引用的 3D 打印实体或组件，记录耗时、盘数、喷嘴和多耗材用量。
- **产品 Product**：可报价对象，由打印件、零件、后处理工时、子产品组成 BOM。
- **报价 Quote**：某一时刻的成本与客户价快照，用于历史追踪和导出。

### 关键业务规则

1. **成本以快照为准**：报价生成后不因后续采购价格变化而自动改变历史报价。
2. **草稿可改，完成锁定**：`draft` 状态允许修改；`completed` 状态默认禁止改 BOM、成本参数和耗材用量，需先解锁或复制为新版本。
3. **库存扣减手动触发**：创建产品和生成报价不扣库存；只有调用 `consume-stock` 或点击按钮才扣库存。
4. **库存不足允许报价，不允许扣减**：报价时可提示库存不足但仍返回成本；扣减时必须校验库存充足。
5. **删除采用软删除优先**：已被采购、打印件、产品或报价引用的数据不得硬删，默认标记为停用/归档。
6. **子产品禁止循环引用**：A 不能引用 A，也不能形成 A→B→C→A 的闭环。
7. **所有金额内部保留 4 位小数，展示四舍五入到 2 位**：避免多层 BOM 计算时累计误差过大。
8. **所有重量统一用克，时间统一用小时，金额统一用人民币元**。

### 待确认业务口径

- 废品损耗是否作用于全部成本，还是只作用于打印耗材与机时成本？当前默认作用于 BOM 小计。
- 报价加成是固定倍率优先，还是支持“最低利润 + 倍率”组合？当前默认固定倍率。
- 后处理是否需要拆为多个工序（打磨/喷漆/组装），还是一期只记录总工时？当前默认一期只记录总工时，二期可拆工序。
- 完成后的产品是否允许生成“实际版”覆盖原产品，还是复制一份实际产品做对比？当前推荐复制/派生实际版，保留预估版。

## 四、功能清单

### 一期（MVP，本次实施范围）

#### 1. 完整 REST API（供 AI 和 Web 调用）
- 耗材 CRUD + 库存查询
- 零件 CRUD（含分类、单位转换）+ 库存查询
- 供应商/采购链接管理
- 采购批次录入 → 自动更新加权平均到手价
- 机器/成本参数配置
- 打印件录入（盘数/耗时/喷嘴/多色用量）→ **API 返回即含打印件成本**
- 产品创建/更新（打印件 + 零件 + 后处理 + 子产品的 BOM）→ **API 返回即含总成本明细树**
- 报价生成（预估/实际两模式）+ 历史查询
- 统计（月度成本/利润、耗材消耗、低库存提醒）
- 健康检查与基础元数据：`GET /api/health`、`GET /api/meta`

#### 2. 后台自动成本计算引擎
- 录入打印件时，自动算：耗材成本（多色分项 × 加权到手价）+ 机时成本（电费 + 折旧）
- 创建/更新产品时，自动算：打印件合计 + 零件（取加权到手价）+ 后处理工时 + 子产品（递归）+ 废品损耗 → 总成本 → 客户报价（× 加成）
- 成本明细树随 API 响应返回，AI 可直接解读告诉用户

#### 3. 仓储系统（核心地基）
- 耗材库：名称、类型（PLA/PETG/ABS…）、颜色、品牌、库存（g）、低库存阈值、加权平均到手价（元/g）
- 零件库：名称、分类（电路板/电源/轴承/螺丝…）、规格、采购单位、使用单位、换算比例、库存数量、低库存阈值、加权平均到手价
- 采购批次记录：每次进货记 数量 + 货款 + 运费 + 日期 + 供应商/链接 → 系统自动算并更新加权平均到手价
- 供应商/链接管理：每个零件保留采购链接与供应商，缺料时一键跳转补货
- 库存预警：低于阈值标红提醒

#### 4. Web 前端（可视化 + 手工兜底）
- 可视化看板：产品列表、库存状态、报价历史、月度统计
- 手工录入/修正：所有 API 支持的操作 Web 上也能做
- 报价单导出：对内（全明细）/对外（仅总价）两套样式，导出图片/PDF
- 采购链接跳转：零件详情页点链接直达供应商

#### 5. 库存联动
- 手动触发扣减：产品完成后调 API 或点按钮，按 BOM 扣减耗材克数、零件数量
- 低库存提醒

#### 6. 基础设置
- 单用户登录占位（本地默认免登录，开关可开启简单口令）
- 数据备份：支持手动导出 SQLite 数据库文件或生成备份包
- 基础初始化：首次启动可创建默认机器、默认成本参数和示例分类

### 二期（预留，不在本次实施）
- 采购优化（"买几个最划算"：运费分摊/满减/阶梯价）
- 切片文件（.gcode/.3mf）解析增强
- 多机器 UI（Web 选择机器）
- 子产品 UI（BOM 编辑时选子产品）
- 费用分摊（批量打印后处理按数量分摊）
- 自动库存扣减（产品完成自动扣，可选配置）
- 多档位报价（零售/批发/VIP）
- 微信小程序上架（uni-app 复用后端 API，云上 HTTPS 域名 + 备案）
- 拓竹云端自动取数（无官方开放 API，作为探索项）

## 五、典型 AI 操作流程示例

**你**："帮我录入这个 MakerWorld 项目 https://makerworld.com.cn/zh/models/2275796"

**AI（Codex/Claude）**：
1. 抓取链接，读出：盘数 4、耗时 9.6h、克重 258g、喷嘴 0.4mm、多色用量
2. 调 `POST /api/print-items`
   ```json
   {
     "name": "小型排风机",
     "plates": 4,
     "print_hours": 9.6,
     "nozzle": "0.4mm",
     "filaments": [
       {"material_id": 3, "grams": 200},
       {"material_id": 5, "grams": 58}
     ]
   }
   ```
3. 平台后台自动算成本，返回：
   ```json
   {
     "id": 12,
     "name": "小型排风机",
     "cost": {"material": 8.54, "machine": 6.24, "total": 14.78}
   }
   ```
4. AI 告诉你："✅ 已录入「小型排风机」打印件，耗材 8.54 元，机时 6.24 元，**打印成本 14.78 元**"

**你**："加上电源、电路板、螺丝，算出完整成本"

**AI**：
1. `GET /api/parts?search=电源` → part_id=8（加权到手价 17.33）
2. `GET /api/parts?search=电路板` → part_id=9（6元）
3. 调 `POST /api/products`
   ```json
   {
     "name": "排风扇成品",
     "mode": "estimate",
     "bom_items": [
       {"kind": "printitem", "ref_id": 12, "qty": 1},
       {"kind": "part", "ref_id": 8, "qty": 1},
       {"kind": "part", "ref_id": 9, "qty": 1},
       {"kind": "part", "ref_id": 10, "qty": 6},
       {"kind": "postprocess", "hours": 0.8}
     ],
     "markup_rate": 1.6
   }
   ```
4. 平台后台算出总成本明细树，返回：
   ```json
   {
     "id": 45,
     "name": "排风扇成品",
     "cost_detail": {
       "printitems": 14.78,
       "parts": 24.13,
       "postprocess": 24.0,
       "subtotal": 62.91,
       "scrap": 3.15,
       "total_cost": 66.06,
       "customer_price": 105.70
     }
   }
   ```
5. AI 告诉你："✅ 已创建产品「排风扇成品」
   📊 成本明细：打印 14.78 + 零件 24.13 + 组装 24.0 + 废品损耗 3.15 = **总成本 66.06 元**
   💰 建议客户报价：**105.70 元**（1.6 倍加成）"

> 整个过程你只发了两句话，AI 完成了抓取、API 调用、成本解读。你随时可在 Web 前端看完整明细或手工改。

### 数据模型

#### Material（耗材）
```python
id: int
name: str                    # 拓竹PLA Basic基础
type: str                    # PLA/PETG/ABS
color: str                   # 黑色
brand: str                   # 拓竹
stock_g: float               # 库存克数
low_stock_g: float           # 低库存阈值
avg_price_per_g: float       # 加权平均到手价（元/克）
updated_at: datetime
```

#### Part（零件）
```python
id: int
name: str                    # 12V电源
category: str                # 电源/电路板/轴承/螺丝...
spec: str                    # 规格说明
purchase_unit: str           # 采购单位（盒/瓶/米）
use_unit: str                # 使用单位（个/克/米）
conversion_ratio: float      # 换算比例（1盒=100个 → 100）
stock_qty: float             # 库存数量（使用单位）
low_stock_qty: float         # 低库存阈值
avg_unit_price: float        # 加权平均到手价（元/使用单位）
default_supplier_id: int | None
updated_at: datetime
```

所有核心表建议统一包含：

金额与数量字段约定：
- 金额内部计算使用 `Decimal`，数据库可存整数分或定点小数；避免用二进制浮点直接做最终金额。
- API 展示金额统一返回 2 位小数，调试明细可额外返回 `raw_amount` 4 位小数。
- 库存数量允许小数，零件也允许按米、克、毫升等非整数单位扣减。
- `avg_price_per_g`、`avg_unit_price` 由采购批次计算得出，不建议前端直接编辑；如需修正，走“库存盘点/价格调整”记录。

常用错误码：
- `VALIDATION_ERROR`：请求字段缺失、类型错误或数值越界。
- `NOT_FOUND`：资源不存在或已归档。
- `LOCKED_RESOURCE`：产品已完成锁定，禁止修改。
- `INSUFFICIENT_STOCK`：库存不足，无法扣减。
- `COST_SOURCE_MISSING`：缺少价格来源，如耗材没有采购批次或到手价为 0。
- `BOM_CYCLE_DETECTED`：子产品引用形成闭环。
- `EXPORT_FAILED`：报价导出失败，需降级为 HTML。

### AI 调用约定

- AI 是外部调用方，不是平台内置模块；平台只暴露 REST API。
- AI 抓取 MakerWorld 页面后的识别结果必须带 `source_url`，便于回溯。
- AI 无法确定的字段应留空或让用户确认，不应随意编造，如喷嘴、颜色拆分、后处理工时。
- API 应返回足够可读的成本明细，使 AI 不需要二次计算就能向用户解释结果。
- AI 批量录入时建议先创建草稿，用户确认后再生成报价或扣减库存。

### 数据一致性与事务

- 创建采购批次、更新库存、重算到手价必须在同一个数据库事务内完成。
- 手动扣减库存必须在同一个事务内完成全部 BOM 扣减；任一项不足则整体失败。
- 产品成本计算可实时计算并写入快照字段；报价以 `Quote.snapshot_json` 为最终历史依据。
- 修改耗材/零件基础信息不应自动改写历史 BOM 的 `unit_price` 与报价快照。
- 删除被引用资源时应返回明确错误，建议使用归档而非硬删除。

### 部署与安全

- 本地默认运行：后端 `http://127.0.0.1:8000`，前端 `http://127.0.0.1:5173`。
- 局域网访问时可绑定 `0.0.0.0`，但建议开启简单口令或反向代理基础认证。
- SQLite 数据库文件需放在明确的数据目录，例如 `backend/data/3dcost.db`，避免被代码清理误删。
- 提供手动备份入口，至少支持复制数据库文件和导出 JSON 备份。
- 后续上云前必须补 HTTPS、域名、备案、访问认证、备份策略和日志脱敏。

> 平台不集成大模型，不含密钥表。所有 AI 交互通过 REST API。

## 七、核心逻辑

### 加权平均到手价（services/pricing.py）
```python
# 例：第一次买 1 个，货款16 + 运费4 = 20 → 到手价 20/个
#     第二次买 2 个，货款28 + 运费4 = 32
#     加权平均 = (20 + 32) / (1 + 2) = 17.33/个
def calc_weighted_avg_price(purchases):
    total_amount = sum(p.goods_amount + p.shipping_fee for p in purchases)
    total_qty = sum(p.qty for p in purchases)
    return total_amount / total_qty if total_qty > 0 else 0.0
```

### 耗材采购入库（卷 → 克）
```python
def create_material_purchase(material_id, qty_rolls, grams_per_roll, goods_amount, shipping_fee):
    purchase = Purchase.create(target_kind='material', target_id=material_id,
                               qty_rolls=qty_rolls, grams_per_roll=grams_per_roll,
                               goods_amount=goods_amount, shipping_fee=shipping_fee)
    # 更新库存
    total_grams = qty_rolls * grams_per_roll
    material = Material.get(material_id)
    material.stock_g += total_grams
    # 重算加权平均到手价（按克）
    all_p = Purchase.filter(target_kind='material', target_id=material_id)
    total_amount = sum(p.goods_amount + p.shipping_fee for p in all_p)
    total_grams_all = sum(p.qty_rolls * p.grams_per_roll for p in all_p)
    material.avg_price_per_g = total_amount / total_grams_all
    material.save()
    return purchase, material.avg_price_per_g
# 示例：买2卷PLA(1000g/卷,80元/卷)+运费10 → 库存+2000g，到手价=(160+10)/2000=0.085元/g
```

### 零件采购入库（盒 → 个）
```python
def create_part_purchase(part_id, qty, goods_amount, shipping_fee):
    part = Part.get(part_id)
    purchase = Purchase.create(target_kind='part', target_id=part_id, qty=qty,
                               goods_amount=goods_amount, shipping_fee=shipping_fee)
    # 更新库存（转换为使用单位）
    part.stock_qty += qty * part.conversion_ratio
    # 重算加权平均到手价（按使用单位）
    all_p = Purchase.filter(target_kind='part', target_id=part_id)
    total_amount = sum(p.goods_amount + p.shipping_fee for p in all_p)
    total_use_qty = sum(p.qty * part.conversion_ratio for p in all_p)
    part.avg_unit_price = total_amount / total_use_qty
    part.save()
    return purchase, part.avg_unit_price
# 示例：买3盒螺丝(100个/盒,15元/盒)+运费5 → 库存+300个，到手价=(45+5)/300=0.167元/个
```

### 打印件成本（services/costing.py）
```python
def calc_print_item_cost(print_item, materials, machine, electricity_price):
    # 耗材成本（多色分项求和）
    material_cost = sum(f.grams * materials[f.material_id].avg_price_per_g
                        for f in print_item.filaments)
    # 机时成本
    depreciation_per_h = machine.price / machine.life_hours
    electricity_per_h = (machine.power_w / 1000) * electricity_price
    machine_cost = print_item.print_hours * (depreciation_per_h + electricity_per_h)
    return {
        "material": round(material_cost, 2),
        "machine": round(machine_cost, 2),
        "total": round(material_cost + machine_cost, 2),
    }
```

### 产品总成本（含子产品递归）
```python
def calc_product_cost(product, bom_items, print_items, parts, labor_rate, scrap_rate):
    printitems_cost = parts_cost = postprocess_cost = subproduct_cost = 0
    for item in bom_items:
        if item.kind == 'printitem':
            printitems_cost += print_items[item.ref_id].cost['total'] * item.qty
        elif item.kind == 'part':
            parts_cost += parts[item.ref_id].avg_unit_price * item.qty
        elif item.kind == 'postprocess':
            postprocess_cost += item.hours * labor_rate
        elif item.kind == 'subproduct':
            subproduct_cost += Product.get(item.ref_id).total_cost * item.qty  # 递归
    subtotal = printitems_cost + parts_cost + postprocess_cost + subproduct_cost
    scrap = subtotal * scrap_rate
    total_cost = subtotal + scrap
    customer_price = total_cost * product.markup_rate
    return {
        "printitems": round(printitems_cost, 2),
        "parts": round(parts_cost, 2),
        "postprocess": round(postprocess_cost, 2),
        "subproduct": round(subproduct_cost, 2),
        "subtotal": round(subtotal, 2),
        "scrap": round(scrap, 2),
        "total_cost": round(total_cost, 2),
        "customer_price": round(customer_price, 2),
    }
```

## 十一、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| MakerWorld 页面结构变化或无法抓取 | AI 不能稳定自动读数 | 平台不依赖抓取器；允许 AI/用户手动录入数据 |
| 采购价格被历史批次改动影响 | 历史报价不可信 | 报价与 BOM 单价保存快照 |
| SQLite 文件损坏或误删 | 数据丢失 | 固定数据目录 + 手动备份 + 后续可加定时备份 |
| 多层 BOM 递归复杂 | 成本算错或死循环 | 单元测试覆盖递归与循环引用检测 |
| 金额浮点误差 | 报价尾差 | 内部使用 Decimal，展示层统一四舍五入 |
| 一期功能膨胀 | 主链路延期 | 坚持先完成阶段 1→6，二期项不插入主链路 |
